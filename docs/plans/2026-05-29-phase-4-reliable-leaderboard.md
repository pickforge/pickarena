# Phase 4 — Reliable Leaderboard Implementation Plan

> **Status:** Draft plan.
> **Parent spec:** `docs/specs/2026-05-29-flutter-benchmark-v2-design.md`
> **Dependencies:** Phase 1 verifier integrity; useful but not strictly dependent on Phase 2/3.

## Goal

Make leaderboard rankings statistically interpretable by adding repeated trials, confidence intervals, cost/time/token reporting, rank uncertainty, and failure taxonomy.

## Success criteria

- Runs can execute multiple trials per model/task.
- Leaderboard shows pass rate with confidence interval.
- Leaderboard shows task/trial count and warns on low sample size.
- Cost, latency, and token summaries are visible.
- Failures are classified into useful categories.
- Export formats include trial and uncertainty data.

## Current code anchors

- `StartRunConfig` and `StartRun` currently have no trial count.
- `RunBloc._onStart` expands task/provider/model into combos and `RetryCombo` deletes rows by run/provider/model/task only.
- `TaskRuns` currently stores aggregate score, tokens, latency, and optional `planId`, but no trial, track, task version, harness, primary pass, or failure tag.
- `LeaderboardRepository` currently ranks by `Dimensions.fromTaskRuns` over aggregate/evaluator scores, not pass-rate statistics.
- `database.g.dart` is generated by Drift and must be regenerated after schema changes.

## Architecture

Current storage treats each task run as a row. Phase 4 should make repeated trials first-class without breaking existing rows.

Recommended model:

```text
Run
  TaskRun
    trialIndex
    taskVersion
    benchmarkTrack
    harnessId
    aggregateScore
    primaryPass
    failureTag
```

Concrete nullable/defaulted columns:

```text
trialIndex INTEGER NOT NULL DEFAULT 0
taskVersion INTEGER NOT NULL DEFAULT 1
benchmarkTrack TEXT NOT NULL DEFAULT 'codegen'
harnessId TEXT NULL
primaryPass BOOL NULL
failureTag TEXT NULL
estimatedCostMicros INTEGER NULL
```

If Phase 2 already added `benchmarkTrack` or `harnessId`, reuse those columns instead of adding duplicates.

Analytics aggregate by:

- model;
- provider/harness;
- task;
- category/tag;
- track;
- trial count.

## Task 1: Add trial configuration

- [ ] Add `trialsPerTask` to `StartRunConfig` and `StartRun`.
- [ ] Expand the run matrix to include trial index.
- [ ] Show total task-run count before starting.
- [ ] Keep default at `1` for current behavior.
- [ ] Add tests for matrix expansion.
- [ ] Include `trialIndex` in progress labels, task-run IDs, retry snapshots, and persisted results.
- [ ] Update `RetryCombo` deletion to target the specific failed trial instead of deleting all rows for the same provider/model/task.

## Task 2: Persist trial and track metadata

- [ ] Add nullable/defaulted fields to `TaskRuns`:
  - `trialIndex`;
  - `taskVersion`;
  - `benchmarkTrack`;
  - `harnessId`;
  - `primaryPass`;
  - `failureTag`.
- [ ] Add database migration.
- [ ] Preserve old rows with safe defaults.
- [ ] Update DAO tests.
- [ ] Increment Drift `schemaVersion` and regenerate `lib/storage/database.g.dart`.
- [ ] Update `TaskRunResult` so the runner passes these fields explicitly instead of deriving them in the DAO.

## Task 3: Define primary pass/fail

- [ ] Add a function that maps evaluator outputs to `primaryPass`.
- [ ] Prefer hidden verifier when present.
- [ ] Fall back to aggregate or existing correctness evaluators for legacy tasks.
- [ ] Store the primary pass decision for fast leaderboard queries.

Suggested precedence:

1. `hidden_test` / hidden verifier evaluator passed;
2. all required public correctness evaluators passed (`compile`, `analyze`, `test`, or task-specific test-author evaluator);
3. legacy fallback: `aggregateScore >= configured pass threshold`.

Never treat missing token/cost/LLM judge data as correctness failure.

## Task 4: Add confidence intervals

- [ ] Implement Wilson interval for pass rates.
- [ ] Add bootstrap helpers for medians if needed.
- [ ] Show `score ± interval` or `low–high`.
- [ ] Add warnings when sample size is too small.

Suggested first pass:

```text
pass_rate = passes / trials
ci = Wilson 95%
```

Test edge cases:

- `0/0` should display unknown, not `0%`;
- `0/n` and `n/n` should still have non-zero interval width;
- one trial should always show a low-sample warning.

## Task 5: Add cost/time/token summaries

- [ ] Store or configure per-model pricing.
- [ ] Estimate cost from prompt/completion tokens when available.
- [ ] Report:
  - median latency;
  - median input tokens;
  - median output tokens;
  - median estimated cost;
  - cost per solved task.
- [ ] Treat missing pricing/token data as unknown, not zero.
- [ ] Keep pricing configuration local/static for the first pass; do not fetch pricing at runtime.
- [ ] Store enough raw token data to recompute cost if pricing changes later.

## Task 6: Add failure taxonomy

Initial tags:

```text
pass
hidden_verifier_failed
public_tests_failed
analysis_failed
compile_failed
regression_failed
harness_timeout
harness_error
no_patch
invalid_output
missed_requirement
flaky_verifier
environment_error
suspected_cheating
unknown
```

- [ ] Add deterministic taxonomy from evaluator/harness outputs.
- [ ] Optionally allow LLM-assisted post-run classification later.
- [ ] Show failure breakdown in model detail view.
- [ ] Store only one primary `failureTag` per task run, using a stable precedence order.
- [ ] Keep raw evaluator details available for deeper debugging.

## Task 7: Update leaderboard UI

- [ ] Add columns for pass rate, CI, trials, median cost, median duration.
- [ ] Keep existing dimension radar as secondary.
- [ ] Add filters for track, task category, difficulty, and harness.
- [ ] Add a low-sample warning when comparing models with insufficient trials.
- [ ] Show per-task variance in model detail.
- [ ] Clearly label aggregate score as legacy/secondary when pass-rate data exists.
- [ ] Avoid ranking rows with unknown `primaryPass` above rows with measured pass/fail data.

## Task 8: Update exports

- [ ] CSV includes trial index, primary pass, failure tag, track, task version, CI fields.
- [ ] Markdown includes leaderboard summary with uncertainty.
- [ ] Preserve compatibility with old run summaries.
- [ ] Include unknown values as empty cells or `unknown`, not `0`.

## Risks

- Repeated trials increase cost quickly; UI must show task-run counts clearly.
- Confidence intervals can be misleading if tasks are not independent.
- Cost estimates depend on accurate provider token reporting.
- Failure taxonomy will start approximate and improve with trajectory data.
- Database migration mistakes can make historical run summaries unreadable; add migration tests before changing leaderboard queries.
- Retrying a failed trial can duplicate or delete rows if trial index is not part of the retry key.

## Exit criteria

Phase 4 is complete when repeated trials produce a leaderboard with pass-rate confidence intervals, cost/time/token summaries, and useful failure breakdowns.

Rollback/compatibility:

- Defaults make old task-run rows appear as `trialIndex=0`, `taskVersion=1`, `benchmarkTrack=codegen`, and unknown primary pass/failure tag until backfilled.
- If the new leaderboard has issues, keep the old aggregate ranking available as a secondary or fallback view.
